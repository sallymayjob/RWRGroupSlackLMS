Part 02

Critical Issues (10 Problems)

01

Phase pages are hollow shells

All 6 pages (01-Initiation through 06-Curriculum-Library) contain identical content — just a nav bar linking to the same databases. They occupy space without adding structure or value. This is project management theater, not content organization.

02

Course Registry doesn't match the 12-month architecture

COURSE-004 = "Month 1" (28 days), COURSE-005 = "Months 2-3" (60 days), COURSE-006 = "Months 4-6" (90 days). Sometimes 1 month = 1 course, sometimes 3 months = 1 course. This makes it impossible to map to the clean current_month = 1-12 model in Firestore.

03

Lessons don't follow ULC format

Your lesson pages use Goal / Key Topics / Activity / Quick Check. But your own Universal Lesson Canvas defines Objective / Core Content / Mission / Verification. The standard you designed isn't in your database.

04

No Month → Module → Lesson hierarchy

The architecture defines Months containing Modules containing Lessons. The Notion DB has flat courses with flat lessons. There's no way to tell which lesson belongs to which module or which month. The checkProgression() function needs this structure to work.

05

Workflow Registry uses killed patterns

WF-002 triggers on "9 AM scheduler" — a time-based delay we explicitly eliminated. WF-004 triggers on "Course completion" but doesn't match the 4-workflow architecture (Course Launcher, Module Engine, Progression Checker, Utility Shortcuts). These entries will confuse anyone reading them.

06

Dashboard numbers don't reconcile

"7% content completion — 24 of 375 lessons written." But the Lessons Registry has 4 entries, and 3 of them are Notion training placeholders ("Databases 101", "Relations: Link Lessons to Assignments"). These aren't real recruiter training lessons. The dashboard tells a story the data doesn't support.

07

Gaps Registry is premature optimization

$6,248 budgeted for gaps like "HRIS Integration" ($385) and "Assessment System" ($550) — with 0 learners tested and 0 lessons validated. You're budgeting for Phase 3 before finishing Week 1. This creates anxiety without driving action.

08

Channel Registry is manual duplication

10 channels manually tracked. But channel names follow a pattern (#course-{id}-{name}) that should be auto-derived from the Course Registry. This is a sync bug waiting to happen.

09

File Registry is admin overhead, not CMS

17 entries tracking deployment guides, technical specs, and SOPs. Learners never see this. It's useful for you, but it doesn't belong alongside curriculum databases. It pollutes the workspace with non-learning content.

10

No Prompt Library, Brand Config, or Weekly Hub

The architecture defined three critical Notion components — a 150-prompt library converted to "Mission Generators," brand configuration per client, and one weekly hub page per module. None of these exist yet.

Part 03

Redesigned Notion Structure

Notion's role in the architecture is Layer 04 — Backend CMS. It stores curriculum, brand configs, and prompt tools. Learners rarely open it. Admins use it to manage content that gets pushed to Slack via Cloud Functions.

🎯
Design principle: Every Notion database must answer a question the system needs. If Slack or Firestore already answers it, Notion shouldn't store it. Notion = content authoring. Firestore = learner state. Slack = delivery.

🔴 Current (8 databases, 6 phases)

  • Course Registry (wrong month grouping)
  • Lessons Registry (4 rows, wrong format)
  • Workflow Registry (outdated triggers)
  • File Registry (admin overhead)
  • Gaps Registry (premature)
  • Channel Registry (manual sync)
  • Assignments Registry (ghost)
  • Resources Library (ghost)
  • 6 hollow phase pages

🟢 Redesigned (5 databases, 3 sections)

  • Courses (master, 1 row per course)
  • Months (12 per course, structured)
  • Lessons (ULC format, linked to months)
  • Prompt Library (mission generators)
  • System Log (workflows + admin ref)
  • Brand Config page (per client)
  • Weekly Hub template (per month)

Database Schemas (Exact Properties)

Every property below maps to something the system actually reads or writes. No vanity columns.

📚

Courses

REDESIGNED
Master course table. One row per deployable course. Replaces old Course Registry with consistent structure.
PropertyTypeNotes
Course ID Title PK — Format: COURSE_12M (one per program)
Title Text Human name, e.g. "RWR 12-Month Recruiter Training"
Audience Select Recruiters | Managers | General | Creators
Total Months Number Always 12 for the primary course
Months Relation → Months DB
Status Select Draft | Writing | Review | Live | Archived
Content Completion Rollup % of months where all lessons are "Ready"
Slack Channel Pattern Text #course-{id}-month{n} — no separate Channel Registry needed
Brand Config Relation Link to brand page (tone, voice, logo) — supports multi-client later
📅

Months

NEW
One row per month per course. This is the missing hierarchy. Maps 1:1 to Firestore's current_month field and the Progression Checker logic.
PropertyTypeNotes
Month ID Title PK — Format: M01 through M12
Course Relation → Courses DB
Month Number Number 1–12. This is what Firestore reads.
Title Text e.g. "Foundations of Recruiting" — learner-facing
Theme Text Monthly learning focus for content coherence
Lessons Relation → Lessons DB
Modules Count Number Typically 4 (Week 1-4). Maps to Firestore current_module 0-3
Daily Lessons Number Micro-lessons per week (default: 5)
Weekly Hub Link URL Link to Notion hub page for deep session (video/slides/quiz)
Status Select Empty | Outlined | Writing | Review | Ready
Lesson Completion Rollup % of lessons in "Ready" status
📝

Lessons

REDESIGNED
Every lesson follows ULC format. This is the actual content table that Cloud Functions read to post Slack threads. One lesson = one Slack thread.
PropertyTypeNotes
Lesson ID Title PK — Format: M01-W1-L01 (Month 1, Week 1, Lesson 1)
Month Relation → Months DB
Week Number 1–4 (maps to module number in Firestore)
Day Number 1–5 (Mon–Fri within the week)
Type Select Daily Micro (5-min text) | Weekly Deep (30-min multimodal)
📌 Objective Text ULC — What the learner will achieve
📌 Core Content Long Text ULC — Minimum 500 words for micro-lessons
📌 Mission Long Text ULC — Action or reflection. Include recommended prompt from Prompt Library.
📌 Verification Text ULC — How completion is proven. Alignment-based, not recall.
Prompt Relation → Prompt Library DB — Linked mission generator
Slack Thread Text Long Text Pre-formatted Slack message (what the bot actually posts)
Status Select Empty | Draft | SOP-5 Review | Ready | Live
SOP-5 Validated Checkbox Passed Course GPT + SOP-5 validation pipeline

NEW

Part 08

Lesson ID Convention (Critical for Automation)

Every lesson needs a deterministic ID that Cloud Functions can parse to know exactly where in the 12-month journey it sits.

M01-W1-L01   = Month 1, Week 1, Lesson 1 (Monday micro)
M01-W1-L02   = Month 1, Week 1, Lesson 2 (Tuesday micro)
M01-W1-L03   = Month 1, Week 1, Lesson 3 (Wednesday micro)
M01-W1-L04   = Month 1, Week 1, Lesson 4 (Thursday micro)
M01-W1-L05   = Month 1, Week 1, Lesson 5 (Friday micro)
M01-W1-DEEP   = Month 1, Week 1, Weekly Deep Session (hub link)
───
M01-W2-L01   = Month 1, Week 2, Lesson 1 ...
... (20 daily + 4 deep = 24 lessons per month × 12 months = 288 total lessons)
💡
This replaces your 375 lesson estimate. The math: 12 months × 4 weeks × (5 daily + 1 deep) = 288 structured lessons. This is achievable. 375 was inflated because months were grouped inconsistently (some had 28 days, some 90). With consistent 4-week months, the number becomes real and trackable.
