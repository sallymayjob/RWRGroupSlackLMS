Part 02

Critical Issues (10 Problems)

01

Phase pages are hollow shells

All 6 pages (01-Initiation through 06-Curriculum-Library) contain identical content — just a nav bar linking to the same databases. They occupy space without adding structure or value. This is project management theater, not content organization.

02

Course Registry doesn't match the 12-month architecture

COURSE-004 = "Month 1" (28 days), COURSE-005 = "Months 2–3" (60 days), COURSE-006 = "Months 4–6" (90 days). Sometimes 1 month = 1 course, sometimes 3 months = 1 course. This makes it impossible to map to the clean current_month = 1–12 model in PostgreSQL.

03

Lessons don't follow ULC format

Your lesson pages use Goal / Key Topics / Activity / Quick Check. But the Universal Lesson Canvas defines Hook / Core Content / Insight / Takeaway / Mission / Verification / Submit Block. The standard you designed isn't in your database.

04

No Month → Week → Lesson hierarchy

The architecture defines Months containing 4 Weeks containing Lessons. The Notion DB has flat courses with flat lessons. There's no way to tell which lesson belongs to which week or which month. The checkProgression() function in n8n Agent 02 needs this structure to fire correctly.

05

Workflow Registry uses killed patterns

WF-002 triggers on "9 AM scheduler" — a time-based delay explicitly eliminated from the architecture. WF-004 triggers on "Course completion" but doesn't match the 14-agent n8n registry. These entries will confuse any developer referencing them.

06

Dashboard numbers don't reconcile

"7% content completion — 24 of 375 lessons written." But the Lessons Registry has 4 entries, and 3 of them are Notion training placeholders ("Databases 101", "Relations: Link Lessons to Assignments"). These aren't real recruiter training lessons. The correct target is 288 lessons (12 months × 4 weeks × 6 sessions). The dashboard tells a story the data doesn't support.

07

Gaps Registry is premature optimization

$6,248 budgeted for gaps like "HRIS Integration" ($385) and "Assessment System" ($550) — with 0 learners tested and 0 lessons validated. You're budgeting for Phase 3 before finishing Week 1. This creates anxiety without driving action.

08

Channel Registry is manual duplication

10 channels manually tracked. But channel names follow a pattern (#course-{id}-{name}) that should be auto-derived from the Course Registry. This is a sync bug waiting to happen.

09

File Registry is admin overhead, not CMS

17 entries tracking deployment guides, technical specs, and SOPs. Learners never see this. It's useful for you, but it doesn't belong alongside curriculum databases. It pollutes the workspace with non-learning content.

10

No Prompt Library, Brand Config, or Weekly Hub

The architecture defined three critical Notion components — a 150-prompt library converted to "Mission Generators," brand configuration per client, and one weekly hub page per module. None of these exist yet.

Part 03

Redesigned Notion Structure

Notion's role in the architecture is Layer 04 — Backend CMS. It stores curriculum, brand configs, and prompt tools. Learners rarely open it. Admins use it to manage content that gets pushed to Slack via n8n Agent 03 (Tutor).

🎯
Design principle: Every Notion database must answer a question the system needs. If Slack or PostgreSQL already answers it, Notion shouldn't store it. Notion = content authoring. PostgreSQL = learner state. Slack = delivery. These three never overlap.

🔴 Current (8 databases, 6 phases)

  • Course Registry (wrong month grouping)
  • Lessons Registry (4 rows, wrong format)
  • Workflow Registry (outdated n8n triggers)
  • File Registry (admin overhead)
  • Gaps Registry (premature)
  • Channel Registry (manual sync)
  • Assignments Registry (ghost)
  • Resources Library (ghost)
  • 6 hollow phase pages

🟢 Redesigned (5 databases, 3 sections)

  • Courses (master, 1 row per course)
  • Months (12 per course, structured)
  • Lessons (ULC format, linked to months)
  • Prompt Library (mission generators)
  • System Log (n8n agents + admin ref)
  • Brand Config page (per client)
  • Weekly Hub template (per month)

Lessons

REDESIGNED
Every lesson follows the 7-section ULC format. This is the content table that n8n Agent 03 (Tutor) reads when a learner runs /learn {lesson_id} to post the lesson as a Slack DM. One lesson = one Slack delivery.
PropertyTypeNotes
Lesson ID Title PK — Format: M01-W1-L01 (Month 1, Week 1, Lesson 1)
Month Relation → Months DB
Week Number 1–4 (maps to w{n}_completed boolean in PostgreSQL progress table)
Day Number 1–5 (Mon–Fri within the week) or DEEP for the weekly session
Type Select Daily Micro (5-min text) | Weekly Deep (30-min multimodal)
📌 Hook Text ULC — 2 sentences. Creates curiosity or challenges an assumption.
📌 Core Content Long Text ULC — 300 words max. Practitioner-level. No filler.
📌 Insight Text ULC — 1 sentence, 50 words max. Specific and counterintuitive.
📌 Takeaway Text ULC — 1 sentence, 15 words max. The one thing to remember.
📌 Mission Long Text ULC — Verb-first action. <5 minutes. Uses tools the learner already has.
📌 Verification Text ULC — 1 question that cannot be answered without completing the mission.
📌 Submit Command Text ULC — Auto-generated: /submit {lesson_id} complete. Appended to every lesson.
Prompt Relation → Prompt Library DB — Linked mission generator
Slack Thread Text Long Text Pre-formatted Slack message (what n8n Agent 03 actually posts via the Slack API)
Status Select Empty | Draft | SOP-05 Review | Ready | Live
SOP-05 Validated Checkbox Passed Content Factory QA pipeline (Claude agents + SOP-05 checklist)

Part 08

Lesson ID Convention (Critical for Automation)

Every lesson needs a deterministic ID that n8n Agent 03 can parse to know exactly where in the 12-month journey it sits, and that n8n Agent 02 can use to write the correct completion flag to PostgreSQL.

M01-W1-L01   = Month 1, Week 1, Lesson 1 (Monday micro) → /submit M01-W1-L01 complete
M01-W1-L02   = Month 1, Week 1, Lesson 2 (Tuesday micro)
M01-W1-L03   = Month 1, Week 1, Lesson 3 (Wednesday micro)
M01-W1-L04   = Month 1, Week 1, Lesson 4 (Thursday micro)
M01-W1-L05   = Month 1, Week 1, Lesson 5 (Friday micro)
M01-W1-DEEP   = Month 1, Week 1, Weekly Deep Session (hub link)
———
M01-W2-L01   = Month 1, Week 2, Lesson 1 ...
... (20 daily + 4 deep = 24 lessons per month × 12 months = 288 total lessons)
💡
This replaces your 375 lesson estimate. The math: 12 months × 4 weeks × (5 daily + 1 deep) = 288 structured lessons. This is achievable. 375 was inflated because months were grouped inconsistently. With consistent 4-week months, the number becomes real and trackable. When a learner runs /submit M01-W1-L05 complete, n8n Agent 02 parses this ID, writes w1_completed = true to PostgreSQL, and calls checkProgression(). The ID format is what makes this deterministic.
